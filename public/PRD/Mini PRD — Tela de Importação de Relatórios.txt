
# 📘 Mini PRD — Tela de Importação de Relatórios (V3)

---

# 1. 🎯 Objetivo da Tela

Permitir que o usuário importe arquivos Excel para:

- alimentar a Base (GRL053)
- alimentar os Layouts Complementares
- atualizar o estado do sistema de forma incremental
- executar automaticamente:
  - matching
  - diagnóstico de divergência

---

# 2. 🧠 Conceito Central

A importação é o ponto onde:

> dados entram → são normalizados → são consolidados → são vinculados → são analisados

Não existe processamento manual após a importação.

---

# 4. 🔄 Tipos de Importação

---

## 4.1 Importação Base (GRL053)

- utiliza Layout Base
- insere ou atualiza registros na Base
- nunca remove registros existentes

---

## 4.2 Importação Complementar

- exige seleção de layout complementar
- insere ou atualiza registros no Complementar
- apenas complementa dados existentes

---

# 6. 🔑 Regra da Chave

```txt
CHAVE = contrato_vinculado + nota_fiscal
````

---

## Regras

* automática
* não configurável
* baseada no tipo da coluna

---

# 7. 🧼 Regra de Normalização (OBRIGATÓRIA)

---

## Contrato vinculado

* remover não numéricos (exceto hífen)
* considerar apenas o primeiro bloco antes do hífen

---

## Nota fiscal

* manter apenas números

---

## Placa

* maiúsculo
* sem espaços
* sem hífen

---

## Regra obrigatória

Nenhum dado entra no sistema sem normalização.

---

## Comentário do ajuste 4 — Regra de chave inválida após normalização

Se após normalização:

* contrato_vinculado resultar vazio
ou
* nota_fiscal resultar vazia

O registro deve:

→ ser descartado
→ NÃO gerar chave_normalizada
→ ser tratado como erro de normalização

---

# 8. 📥 Regra de Importação da Base (CRÍTICA)

---

## Modelo

Incremental acumulativo

---

## Para cada linha:

* chave não existe → INSERIR
* chave existe:

  * comparar os dados normalizados relevantes
  * se houver qualquer diferença → ATUALIZAR
  * se NÃO houver diferença → IGNORAR

---

## Comentário do ajuste 3 — Campos relevantes para comparação

Campos relevantes para comparação:

* contrato_vinculado
* nota_fiscal
* placa_normalizada

Campos analíticos (peso, data, clifor, etc):
* NÃO devem disparar atualização
* são apenas informativos

---

## Comentário do ajuste 1 — Base de comparação obrigatória (Base)

A comparação deve ser feita EXCLUSIVAMENTE com base nos dados já normalizados.

É proibido utilizar:
* dados_originais completos
* valores brutos do Excel
* strings sem normalização

A comparação deve considerar apenas os campos estruturais já normalizados.

---

## Regras críticas

```txt
Importação NÃO substitui a base.
Importação NÃO remove registros.
Importação NÃO apaga dados anteriores.
```

---

## Comportamento esperado

* base cresce ao longo do tempo
* não há perda de dados
* cargas parciais são suportadas

---

# 9. 📥 Regra de Importação do Complementar

---

## Para cada linha:

* chave não existe na base → IGNORAR
* chave existe:

  * não existe → INSERIR
  * existe:

    * comparar os dados normalizados relevantes
    * se houver qualquer diferença → ATUALIZAR
    * se NÃO houver diferença → IGNORAR

---

## Comentário do ajuste 1 — Base de comparação obrigatória (Complementar)

A comparação deve ser feita EXCLUSIVAMENTE com base nos dados já normalizados.

É proibido utilizar:
* dados_originais completos
* valores brutos do Excel
* strings sem normalização

A comparação deve considerar apenas os campos estruturais já normalizados.

---

## Regras críticas

* complementar NÃO cria base
* complementar NÃO altera contrato da base
* complementar NÃO remove dados

---

# 10. 🔗 Regra de Matching

Após importação:

* existe na base + complementar → Vinculado
* existe apenas na base → Aguardando

---

## Comentário do ajuste final 1 — Ordem obrigatória de processamento

Ordem obrigatória de processamento:

1. Executar matching principal utilizando:
   contrato_vinculado + nota_fiscal

2. Apenas se NÃO houver match:
   executar diagnóstico secundário (nota + placa)

Regras:

* É proibido executar diagnóstico secundário antes do matching principal
* É proibido misturar critérios de matching com diagnóstico

---

# 11. 🔎 Diagnóstico Secundário

Quando não houver match por chave principal:

```
Nota + Placa
```

---

## Comentário do ajuste 1 — Condição obrigatória para diagnóstico secundário

O diagnóstico secundário (Nota + Placa) só deve ser executado quando:

* a nota fiscal for válida
* a placa for válida

Considera-se placa válida quando:
* não é nula
* não é vazia
* após normalização possui pelo menos 1 caractere

Caso a placa seja inválida ou ausente:
→ NÃO executar diagnóstico secundário
→ manter status como "Aguardando"

---

## Resultados

* 1 candidato:

  * contrato diferente → Divergente
* múltiplos → Ambíguo
* nenhum → Aguardando

---

## Regra crítica

```txt
Placa NÃO faz parte da chave.
```

---

# 12. 🔁 Reprocessamento

* automático após importação
* NÃO existe botão manual

---

## Comentário do ajuste final 2 — Reprocessamento após alteração

Se um registro já vinculado sofrer alteração em dados relevantes:

→ o registro deve ser reprocessado automaticamente
→ o status deve ser recalculado

Regras:

* o sistema NÃO deve manter status antigo após alteração
* o status deve sempre refletir o estado atual dos dados

---

## Regra de performance

* reprocessar apenas chaves afetadas

---

# 13. 🚨 Validações

Bloquear importação quando:

* layout inválido
* ausência de contrato vinculado
* ausência de nota fiscal
* estrutura incompatível

---

## Comentário do ajuste 8 — Regra de erro na normalização

Se não for possível normalizar:

* contrato_vinculado
ou
* nota_fiscal

O registro deve:
→ ser ignorado
→ ser contabilizado como erro
→ NÃO ser persistido

---

## Comentário do ajuste 2 — Separação de tipos de não processamento

Tipos de não processamento:

* ignorado_sem_base:
  quando a chave não existe na Base

* ignorado_sem_alteracao:
  quando o registro já existe e não houve mudança

* erro_normalizacao:
  quando não foi possível normalizar contrato ou nota

Esses tipos devem ser tratados separadamente no processamento,
mesmo que a interface exiba apenas como "ignorados".

---

## Comentário do ajuste 6 — Definição de registro válido

Um registro é considerado válido quando:

* possui contrato_vinculado válido
* possui nota_fiscal válida
* foi normalizado com sucesso
* possui chave_normalizada válida

Caso qualquer condição falhe:

→ o registro NÃO deve ser persistido
→ o registro NÃO participa do sistema

# 16. 🔒 Regra de Concorrência

* apenas 1 importação por vez
* bloquear importações simultâneas

---

# 17. 🚫 Regras Estruturais

O sistema NÃO deve:

* duplicar registros
* excluir automaticamente
* inferir dados
* depender da ordem de importação

---

## Comentário do ajuste 4 — Idempotência da importação

A importação é idempotente.

Importar o mesmo arquivo múltiplas vezes NÃO deve:
* gerar novos registros
* atualizar registros existentes
* alterar o estado da conferência
