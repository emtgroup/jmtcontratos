# 📘 Mini PRD — Tela de Layouts Complementares

## Objetivo
Permitir mapear colunas de cada relatório complementar para tipos do sistema, mantendo compatibilidade com a Base.

## Conceito central
Cada layout complementar adapta o arquivo de origem ao modelo do sistema; a lógica de vínculo continua centralizada no PRD principal.
O algoritmo detalhado de matching/diagnóstico que usa esses layouts está no `PRD — Motor de Matching e Diagnóstico.md`.

## Regras obrigatórias

### 1) Chave idêntica à Base
- `CHAVE = contrato_vinculado + nota_fiscal`
- Não existe chave alternativa por layout.

### 2) Tipagem obrigatória
- Tipos estruturais obrigatórios e únicos por layout:
  - Contrato vinculado
  - Nota fiscal
- Tipos analíticos opcionais: placa, peso líquido, data, clifor, outros.

### 3) Validações de salvamento
Bloquear salvamento se:
- faltar contrato vinculado;
- faltar nota fiscal;
- houver duplicidade dos tipos estruturais;
- nome de coluna vazio;
- nome de coluna duplicado.

### 4) Regra de vínculo
- Complementar só vincula se a chave existir na Base.
- Sem chave na Base: registro ignorado para conferência e não persistido operacionalmente em `registros_complementares` (apenas contabilizado como `ignorado_sem_base` no resumo/log).

### 5) Regra da placa
- Placa não participa da chave.
- Placa não define vínculo principal.
- Uso exclusivo para diagnóstico secundário (`nota + placa`) quando elegível.

### 6) Regra dos dados analíticos
- Peso, data, clifor etc. são informativos.
- Não alteram status por si só.

### 7) Versionamento
- Não há versionamento de layout.
- Configuração ativa substitui a anterior.

## Fluxo
1. Usuário cria/edita layout complementar.
2. Sistema valida mapeamento mínimo.
3. Layout é habilitado para importação complementar.
4. Importações usam layout para normalizar e consolidar por chave.

## Edge cases
- Layout complementar válido, mas sem coluna de placa: vínculo principal continua possível; diagnóstico secundário pode ficar indisponível.
- Mesmo registro presente em múltiplos layouts para a mesma chave: status final deve ser `ambiguo`, com `origem = NULL`, sem priorização automática e sem layout preferido.
- Arquivo com cabeçalho divergente do layout: erro explícito de estrutura.

## Proibições
- Alterar lógica de chave no layout.
- Criar vínculo sem chave válida.
- Alterar dados da Base a partir do complementar.
- Inferir mapeamento automático sem regra explícita.

## Resultado esperado
- Layouts complementares consistentes entre si e com a Base.
- Importação complementar previsível e rastreável.
- Suporte correto ao matching e ao diagnóstico secundário.
